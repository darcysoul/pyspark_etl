# ETL-процесс обработки данных интернет-магазина (PySpark)

## Описание проекта

Проект реализует ETL-пайплайн на **PySpark** для обработки данных интернет-магазина с фокусом на очистку данных, построение аналитических витрин и базовую оптимизацию выполнения.

Пайплайн моделирует реальный сценарий работы с «грязными» источниками данных и демонстрирует полный цикл их обработки — от загрузки до формирования готовых витрин для аналитики.

## Основной функционал

* Загрузка и обработка нескольких источников данных:

  * `users`, `products`, `orders`, `order_items`
* Использование явных схем при чтении данных
* Очистка данных:

  * удаление дубликатов заказов
  * фильтрация `null` в ключевых полях
  * удаление некорректных значений (`quantity <= 0`)
  * исключение отменённых заказов
  * удаление «битых» связей между таблицами
* Обогащение данных через `join`
* Построение витрин:

  * дневная выручка по странам
  * продажи по категориям
  * агрегаты по пользователям
* Сохранение результатов в формате **Parquet**
* Партиционирование данных по дате
* Базовые проверки качества данных
* Анализ производительности Spark-пайплайна

## Архитектура

Пайплайн реализован по классической схеме обработки данных:

* **Raw слой** — исходные CSV-файлы
* **Processed слой** — очищенные и объединённые данные
* **Data marts** — агрегированные витрины для аналитики

---

### Поток обработки данных

1. **Загрузка данных**

   * чтение CSV с явным описанием схемы
   * приведение типов

2. **Очистка и валидация**

   * удаление дубликатов (`order_id`)
   * фильтрация некорректных значений
   * проверка ссылочной целостности

3. **Трансформация**

   * объединение таблиц (`join`)
   * расчёт метрик (выручка, количество товаров)

4. **Агрегация**

   * построение витрин через `groupBy` и `agg`

5. **Сохранение**

   * запись в Parquet
   * `partitionBy("date")` для витрин

## Витрины данных

### Daily Sales Mart

Гранулярность: **день × страна**

* количество заказов
* количество покупателей
* количество товаров
* выручка

### Category Sales Mart

Гранулярность: **день × категория**

* количество заказов
* количество товаров
* выручка

### User Sales Mart

Гранулярность: **пользователь**

* количество заказов
* выручка
* первый и последний заказ

## Проверки качества данных

В пайплайне реализованы проверки:

* уникальность `order_id` после очистки
* отсутствие `null` в ключевых полях витрин
* `revenue >= 0`
* отсутствие несвязанных записей:

  * `orders` без пользователя
  * `order_items` без товара
* непустой результат витрин

Проверки выполняются через `assert` и/или логирование перед записью данных.

## Оптимизация и производительность

Реализованы базовые подходы к оптимизации:

* анализ физического плана выполнения (`explain("formatted")`)
* выявление операций `shuffle`
* использование **broadcast join** для небольших таблиц
* партиционирование по дате для ускорения чтения
